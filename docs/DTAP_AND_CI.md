# DTAP and CI Evidence

This project uses a GitHub Actions development lane in:
- `.github/workflows/ci.yml`
- `.github/workflows/acceptance.yml`

## Development (D)
- Developers work in feature branches and run local compile/tests.
- `develop` is the integration branch for day-to-day development.
- CI runs quality gates for pull requests and pushes.

## Testing (T)
Testing lane is represented by the `test` branch and Linux CI quality job:
- Java compile for source and tests.
- Unit test execution via `AllTests`.
- Code coverage report generation (JaCoCo: HTML/XML/CSV).
- Coverage percentage summary output (`coverage-summary.txt`).
- Line coverage quality gate (workflow fails if line coverage < 75%).
- Static quality report (`javac -Xlint`) exported as artifact.

## Acceptance (A)
- Acceptance lane is represented by the `acceptance` branch.
- A dedicated acceptance workflow (`.github/workflows/acceptance.yml`) runs on `windows-latest` to simulate acceptance in an environment different from Linux:
  - Compiles source and tests.
  - Runs `AllTests`.
  - Builds runnable JAR artifact.
- This supports lecturer guidance to validate behavior on another runner/OS before release.

## Production (P)
- Production lane is represented by `main`.
- On push to `main`, CI builds and publishes a production release JAR artifact.
- Full deployment to a live production environment is outside current project scope.

## CI Artifacts
Generated by workflow:
- `coverage-html/`
- `coverage.xml`
- `coverage.csv`
- `coverage-summary.txt`
- `lint-report.txt`
- `release/JabberPoint.jar` (Linux quality build)
- `acceptance-windows-artifacts` (Windows acceptance simulation)
- `production-release-artifact` (main branch release)
